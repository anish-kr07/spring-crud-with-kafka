Spring CRUD — Interview Concept Guide

Compiled from the running spring-crud codebase (Spring Boot 4 + Java 26 + JPA + H2 + Spring Security). Each section answers a question an interviewer is likely to ask.

1.1 Connection pool — why Hikari, why not size=1000

Why we need a pool at all. Opening a database connection is expensive: TCP handshake, TLS negotiation, auth, server-side process/thread allocation (Postgres forks a process; MySQL spawns a thread), session state setup. Easily 5–50 ms per open. A pool keeps a small set of connections warm and hands them out for the duration of one query/transaction, then returns them — typical acquire is sub-millisecond.

Why HikariCP specifically. Spring Boot 2+ ships it as the default. It's the fastest mainstream JDBC pool (lock-free fast path, ConcurrentBag), small (~130 KB jar), and battle-tested. You almost never need to swap it out.

Interview gold — "Why can't we just set maximum-pool-size=1000?"

Because the bottleneck is the database, not the pool. The DB can't usefully process 1000 queries in parallel — and trying to do so makes things slower, not faster.

  1. DB-side cost per connection. Each connection is an OS-level resource on the database. Postgres: a separate backend process, ~5–10 MB RAM each, plus per-session work_mem. MySQL: a thread per connection. 1000 connections → multiple GB of DB RAM gone before a single query runs.
  2. Contention beats parallelism past a point. CPU cores are finite. Beyond ~2× cores worth of active connections, you're context-switching more than computing. Lock contention (row locks, latches, buffer-pool pages) climbs sharply.
  3. Little's Law. Throughput = concurrency / latency. If the DB can sustain 100 queries/sec at 10 ms each, raising concurrency to 1000 doesn't increase throughput — it just inflates the queue, pushing latency to 10 seconds.
  4. Storage saturates. Disk I/O queue depth and IOPS budget are fixed. More writers contending for the same WAL/redo log = more waiting.
  5. The pool stays cheap, but it doesn't help. Hikari can hand out 1000 connections in microseconds; the DB still processes them serially-ish at the bottleneck.

The HikariCP wiki ("About Pool Sizing") cites a Postgres benchmark where throughput peaks around 2× CPU cores + spindle count, then drops as you add more connections. The starting formula:

connections = ((core_count * 2) + effective_spindle_count)

For a typical 8-core DB on SSD: ~17 connections. Total across all app instances, not per instance.

Sizing rules of thumb

  • Total connections across all app instances should be ≤ DB's max_connections, leaving headroom for migrations/admin tools. 5 app instances × pool size 10 = 50 connections. Postgres default is 100 — you're at 50% before autoscaling.
  • Match pool size to the bottleneck. If the DB is CPU-bound, ~(2 × cores). If I/O-bound (lots of disk waits), a bit higher. Measure, don't guess.
  • Watch for pool-saturation alerts: HikariPool-1 - Connection is not available, request timed out after 30000ms. Means raise pool size or fix slow queries — usually the latter.

2. JPA Entities

2.1 Lombok on entities — @Data is a TRAP

Use @Getter + @Setter, NOT @Data on JPA entities.
  • @Data generates equals/hashCode from all fields.
  • Before save(), id is null; after save, it changes — entity in a HashSet "disappears".
  • equals/hashCode may touch lazy associations → surprise SQL or LazyInitializationException.
Safer: id-based equals or skip them entirely (what we do).

2.2 Annotations

AnnotationNotes
@Id + @GeneratedValue(strategy=IDENTITY)DB auto-increment. Hibernate cannot batch inserts with IDENTITY (it needs the generated id back per row). For bulk inserts prefer SEQUENCE with allocationSize>1.
@Column(nullable=false, unique=true, length=...)Explicit DDL. Survives field renames.
@TransientJPA ignores this field/getter — not persisted. Different from Java's transient keyword (serialization).
java.time.LocalDateHibernate 6+ handles java.time natively. No @Temporal needed (that was for java.util.Date).
MoneyNever use float/double. Use BigDecimal.

3. Repositories

  • JpaRepository<Entity, Id> gives save, findById, findAll, delete, count via runtime proxy.
  • Derived queries: method name parsed at startup — findByEmail, findByDepartmentAndLastName, countByDepartment, existsByEmail.
  • JPQL via @Query: use entity names (Employee, not employees), aliases (select e from Employee e), :paramName binding with @Param.
Interview gold — findById vs getReferenceById:
  • findById(id) — issues SELECT, returns full entity (or empty Optional).
  • getReferenceById(id) — no SELECT, returns a lazy proxy. Perfect for FK assignment where you only need the id.
Pair getReferenceById with existsById upfront to get a clean 404 instead of a deferred EntityNotFoundException.

4. DTO pattern

  • Never return entities from controllers. Reasons: lazy-loading explosions, leaks internal fields (version, audit), couples API to schema.
  • Separate EmployeeRequest (inbound) from EmployeeResponse (outbound). Response can carry computed/derived fields the client never sends (e.g. fullName).
  • Use static factories (EmployeeResponse.from(entity)) — manual mapping is 5 lines and shows you understand the why.
  • Mapping libraries: MapStruct (compile-time, fast) > ModelMapper (reflection, slow, surprising — avoid).

7. Service layer

  • @Service = stereotype of @Component for business logic.
  • Constructor injection with final fields. No @Autowired when there's a single constructor — Spring picks it up automatically.
  • @Transactional(readOnly=true) at class level for all reads. Override on writes with bare @Transactional.

7.1 Dirty checking — why no save() on update

Most-asked transaction question. 
@Transactional
public void update(Long id, EmployeeRequest req) {
    Employee e = repo.findById(id).orElseThrow(...);
    e.setFirstName(req.getFirstName());
    // no save() needed!
}
Why it works:
  1. findById returns a managed entity tracked by the EntityManager.
  2. Hibernate keeps a snapshot of original field values in the persistence context.
  3. At commit, Hibernate auto-flushes, compares fields to snapshot, issues UPDATE for dirty fields.
Without @Transactional: findById opens its own short tx, returns a detached entity, your setters mutate plain Java, no UPDATE is issued — silent bug.

7.2 Self-invocation pitfall

Calling a @Transactional method from another method in the same class bypasses the proxy → no transaction. Solution: split into another bean, or use AopContext.currentProxy().

7.3 Rollback rules

  • Spring rolls back only on RuntimeException and Error by default.
  • Checked exceptions commit unless you add @Transactional(rollbackFor = MyChecked.class).

8. Exception handling — @RestControllerAdvice + ProblemDetail

  • @RestControllerAdvice = @ControllerAdvice + @ResponseBody. Spring routes any controller exception through its @ExceptionHandler methods, picking the most specific match.
  • ProblemDetail (Spring 6 / Boot 3+, RFC 7807): built-in. Sets application/problem+json content type automatically. Use setProperty for extension fields.
ExceptionHTTPTriggered by

Why custom NotFoundException extends RuntimeException:
  • Unchecked → no throws on every signature; works in lambdas (orElseThrow(...)).
  • Spring rolls back automatically (RuntimeException).
  • Decouples service from HTTP — service throws domain exception, advice maps to status. (Avoid throw new ResponseStatusException(...) in services for the same reason.)

9. JPA relationships — @OneToMany / @ManyToOne

9.1 Owning vs inverse side

Only ONE side owns the relationship — the side that holds the foreign key column.

SideAnnotationOwning?
Employee.department (FK lives here)@ManyToOne @JoinColumn(name="department_id")OWNING
Department.employees@OneToMany(mappedBy = "department")INVERSE
Forgetting mappedBy — JPA assumes both sides own → creates a junction table department_employees. Extra table, extra inserts. Always declare mappedBy on the inverse side.

9.2 Fetch defaults

@ManyToOneEAGER ❌ (the bad default)
@OneToManyLAZY ✅
@OneToOneEAGER ❌
@ManyToManyLAZY ✅

Always declare fetch = LAZY on @ManyToOne. Override per-query with @EntityGraph or JOIN FETCH when needed.

9.3 Cascade and orphanRemoval

  • Cascade lives on the parent's collection (Department.employees), not on the child.
  • CascadeType.REMOVE — deletes children when parent is deleted.
  • orphanRemoval = true — also deletes a child when it's removed from the collection (even without deleting the parent).

9.4 The N+1 problem

The single most-asked JPA interview question.

Naive: departmentRepo.findAll() + loop accessing dept.getEmployees().size():

1 query : SELECT * FROM departments
N queries: SELECT * FROM employees WHERE dept_id=?  -- per department
For 100 departments → 101 queries.

Three fixes (know all three):

  1. @EntityGraph(attributePaths = "employees") on the repository method — declarative, JPA standard.
  2. JOIN FETCH in JPQL: select d from Department d left join fetch d.employees.
  3. @BatchSize(size=N) on the collection — Hibernate-specific; loads lazy collections in batches.

9.5 LazyInitializationException

  • Touching a lazy field outside the persistence context throws LazyInitializationException.
  • Triggered most often when controllers serialize entities (avoid this — use DTOs).
  • spring.jpa.open-in-view=false surfaces these bugs early in dev.

10. Spring Security basics

  • Adding spring-boot-starter-security to build.gradle auto-secures every endpoint with HTTP Basic.
  • Default username = user, password = a generated UUID logged at startup.
  • Set spring.security.user.name / spring.security.user.password in application.properties to fix the password.
  • For real apps, write a SecurityFilterChain bean and define rules with authorizeHttpRequests. Add @Configuration + @EnableWebSecurity on the class.
  • Disable CSRF for stateless APIs (csrf.disable()). Do NOT disable it for browser-based form apps.

12. Things to revisit

  • Pagination: Pageable, Page<T> vs Slice<T> (Slice avoids the COUNT query).
  • Caching: @EnableCaching, @Cacheable, @CacheEvict. Default ConcurrentMap works in dev; swap to Caffeine in prod.
  • Auditing: @EnableJpaAuditing, @CreatedDate, @LastModifiedDate, @EntityListeners(AuditingEntityListener.class).
  • Optimistic locking: @Version Long version → throws OptimisticLockingFailureException on concurrent updates.
  • Tests: @DataJpaTest (repo slice), @WebMvcTest (controller slice with @MockitoBean), @SpringBootTest (full context).
