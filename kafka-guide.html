Kafka — Interview Concept Guide

Compiled from the spring-crud codebase: local Kafka via docker-compose, Spring Kafka producer/consumer, and the transactional outbox pattern.

1. Why Kafka in this project

The synchronous REST API tells the client "I created the employee." We also want to react to that change asynchronously — projections, search indexing, notifications, analytics. Doing all of that in the request thread couples them to API latency and reliability.

Kafka decouples by being a durable log: one writer, many independent readers, retention measured in days. The app writes once; downstream services react on their own schedules without the app knowing about them.

2. Cluster topology — KRaft, no Zookeeper

  • Kafka 4.x runs in KRaft mode — internal Raft consensus replaces Zookeeper for cluster metadata.
  • In a single-node dev cluster (our setup), the same broker process plays both controller and broker roles.
  • Image used: apache/kafka:3.9.0 (the official ASF image).

2.1 Multi-listener config — KAFKA_LISTENERS

Brokers expose named listeners so different network paths get their own socket and security policy.

KAFKA_LISTENERS = "PLAINTEXT://:9092,CONTROLLER://:9093,INTERNAL://:9094"
ListenerPortUsed by
PLAINTEXT9092Spring app on the host (mapped via compose ports:)
CONTROLLER9093KRaft quorum traffic only — never clients
INTERNAL9094Other containers on the docker network (Kafka UI)

Two paired settings:

  • KAFKA_LISTENERS — what sockets the broker binds.
  • KAFKA_ADVERTISED_LISTENERS — what address the broker tells clients to use for follow-up traffic. Different per listener: PLAINTEXT://localhost:9092 for host clients, INTERNAL://kafka:9094 for container clients (docker DNS).
Classic trap: "My consumer in a container can't reach Kafka, but my host consumer can." The broker advertises only localhost:9092. Inside a container, localhost means the container itself. Fix: add an INTERNAL listener and advertise the docker service name.

3. Topic vs Aggregate vs Event type — the three layers

This is the most-confusing-at-first concept set. Three different things at three different layers:

ConceptLayerAnswersExample
TopicTransport (Kafka)"What channel does this travel on?"employee.events
Aggregate typeDomain (your code)"What kind of thing is this event about?"Employee
Aggregate idDomain (your code)"Which one?"42
Event typeDomain (your code)"What just happened to it?"EmployeeCreated

3.1 Why one topic per aggregate, NOT one per event type

Naive design: employee_created, employee_updated, employee_deleted as separate topics. Industry-standard: one employee.events topic carrying all event types, with the type in a header.

Reasons to use one-topic-per-aggregate:
  1. Per-aggregate ordering. All events for employee 42 use "42" as the message key → hash to the same partition → strict order. With separate topics, "Created" and "Updated" are on different partitions and there's no cross-topic ordering; consumers may see Updated before Created.
  2. Fewer topics, simpler ops. Each topic is its own retention/replication/ACL config. 50 event types × per-type topics = 50 sets of decisions.
  3. New event types are free. Add EmployeePromoted: just publish. No broker change.
  4. Consumers want "all changes to this aggregate." A read-model projection subscribes once.
  5. Per-type topics encourage anemic streams. One channel per aggregate keeps "everything that happened" coherent.
Exceptions where per-event-type topics make sense: cross-domain pub/sub with very different consumers, different retention needs (snapshot vs delta), different ACLs.

3.2 Anatomy of a single message

topic:     "employee.events"
partition: 3                          ← derived from hash(key) % num_partitions
offset:    9817                       ← position within the partition
key:       "42"                       ← the aggregate id - controls partitioning
headers:
  event-type:     "EmployeeCreated"   ← what happened
  aggregate-type: "Employee"          ← which kind of thing
  outbox-id:      "17"                ← for consumer dedup
value:     {"id":42, "firstName":"Hedy", ...}    ← payload (DTO as JSON)

A consumer dispatches by reading the header:

switch (headerValue(record, "event-type")) {
    case "EmployeeCreated" -> ...
    case "EmployeeUpdated" -> ...
    case "EmployeeDeleted" -> ...
}

4. Partitions, keys, and consumer groups

ConceptNotes
PartitionA topic's unit of parallelism. Each partition is a strictly-ordered append-only log. Order is guaranteed within a partition, never across partitions.
Message keyRouting input. Default partitioner: hash(key) % numPartitions. Same key → same partition → ordered. null key → round-robin (no ordering).
Consumer groupA set of consumers that share the topic's partitions. Members of the same group split the work; members of different groups each get the full stream (pub/sub fan-out).
OffsetPer-partition position. Consumer commits offsets to remember "I've read up to here." On restart, it resumes from the committed offset.
RebalanceWhen a consumer joins/leaves the group, partitions are reassigned. Brief pause in consumption.
Choosing partition count: Cannot decrease later (only increase, and that breaks key→partition stability). Pick a number that supports future scale. Common heuristic: target throughput per partition × number of consumers in the largest group.

4.1 Consumer group dynamics — what happens with multiple listener beans

This project has two listener beans (EmployeeEventListener and DepartmentEventListener) both with groupId="spring-crud". Each becomes a separate consumer instance in the group at runtime. The way Kafka splits work between them depends on which topics they subscribe to:

SetupResult
Two listeners, same group, same topicKafka assigns each partition to exactly ONE consumer in the group. With a single-partition topic, only one listener gets messages — the other sits idle. Classic foot-gun.
Two listeners, same group, different topicsEach listener owns its topic's partitions. No conflict. ✅ This is our setup: Employee listener gets employee.events, Department listener gets department.events.
Two listeners, different groups, same topicBoth listeners receive every message (pub/sub fan-out). Use this when one logical app needs two independent reactions to the same stream.
Diagnosing this in logs. When the app starts you'll see lines like: 
[Consumer-1] Finished assignment for group at generation 24:
  {consumer-spring-crud-1=Assignment(partitions=[department.events-0]),
   consumer-spring-crud-2=Assignment(partitions=[employee.events-0])}
Each listener's assigned partitions are listed. If a listener has empty assignment, it won't fire. Check this when "my consumer isn't getting messages" mysteries hit.

5.1 Sync vs async send

kafkaTemplate.send(...) returns a CompletableFuture. By default failures are silent unless you attach a callback. Two patterns:

PatternCodeWhen to use
Async + callbacksend(...).whenComplete((r, ex) -> ...)Read-path / observability events where the request mustn't wait for Kafka. Failures are logged and dropped. Used by DepartmentEventPublisher.
Syncsend(...).get(timeout)State-change events that must be durably acked before "moving on" (mark outbox row as published). Used by OutboxRelay.

5.2 Why this project has its own KafkaConfig instead of relying on auto-config

Spring Boot auto-configures KafkaTemplate<?, ?> (generic wildcard). In Spring DI, generics are invariant: KafkaTemplate<?, ?> is NOT assignable to KafkaTemplate<String, String>. So a constructor that asks for the typed form fails:

Parameter 1 of constructor in OutboxRelay required a bean of type
'org.springframework.kafka.core.KafkaTemplate' that could not be found.

Fix in kafka/KafkaConfig.java: declare typed ProducerFactory<String, String>, KafkaTemplate<String, String>, ConsumerFactory<String, String>, and ConcurrentKafkaListenerContainerFactory<String, String> beans explicitly. @EnableKafka on the same class activates @KafkaListener scanning.

Bonus benefit: producer/consumer settings live in one auditable Java file rather than scattered across application.properties. In a real codebase that grows, the explicit config is easier to diff and review.

6.1 Auto vs manual ack

  • Auto (default): consumer commits offsets periodically. If the listener method throws after committing, that message is lost.
  • Manual: declare an Acknowledgment param on the listener; call ack.acknowledge() only after successful processing. Safer for at-least-once.

7. Transactional outbox pattern (the textbook event publisher)

7.1 The problem it solves

You want to publish a Kafka event when an Employee is created. The naive code:

@Transactional
public void create(...) {
    employeeRepo.save(emp);
    kafkaTemplate.send("employee.events", emp);   // ← THE BUG
}
Failure modes:
  • Kafka is down → send throws → DB rolls back. Business write fails because of an unrelated downstream system. ❌
  • Kafka accepts the message but the DB tx rolls back later → "ghost" event for an employee who doesn't exist. ❌
  • App crashes between save and send → DB has the row, Kafka doesn't → silent data divergence. ❌
This is the dual-write problem: writing to two systems atomically with no shared transaction.

7.2 The pattern

  1. In the same DB transaction as the business write, INSERT a row into an outbox_events table.
  2. Both rows commit atomically — same DB, same tx. No dual-write.
  3. A separate background poller reads unpublished outbox rows and pushes them to Kafka.
  4. On successful publish, the poller marks the row as published.
POST /v1/employees
   └─ EmployeeService.create()  [@Transactional]
        ├─ employeeRepo.save(e)              ← INSERT employees
        └─ outboxPublisher.record(...)       ← INSERT outbox_events
                                               (atomic — same tx)

every 1s:
   OutboxRelay.publishPending()  [@Transactional]
        ├─ findUnpublishedForUpdate(50)      ← SELECT FOR UPDATE
        ├─ kafkaTemplate.send(...).get()     ← wait for broker ack
        └─ event.setPublishedAt(now())       ← UPDATE outbox_events

EmployeeEventListener — separate consumer thread
        └─ logs the event

7.3 Outbox table shape

ColumnPurpose
idPK (auto-increment)
aggregate_type"Employee"
aggregate_id"42" — used as Kafka message key
event_type"EmployeeCreated", "EmployeeUpdated", ... — sent as Kafka header
payloadJSON String (serialized DTO)
created_atFor ordering
published_atNULL = not yet sent. Indexed: (published_at, created_at)

7.4 Key implementation details

Propagation.MANDATORY on the publisher

@Transactional(propagation = Propagation.MANDATORY)
public void record(...) { ... }

Asserts that this method must be called from within an existing transaction. If the caller forgot @Transactional, this throws — failing loud rather than silently breaking atomicity.

Pessimistic lock on the relay's read

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select e from OutboxEvent e where e.publishedAt is null order by e.createdAt asc")
List<OutboxEvent> findUnpublishedForUpdate(Limit limit);

Generates SELECT ... FOR UPDATE. Without it, two poller instances would race and double-publish.

SKIP LOCKED — the multi-instance upgrade.
Plain FOR UPDATE serializes pollers (one waits for the other). The textbook fix is SELECT ... FOR UPDATE SKIP LOCKED — pollers grab different rows, run in parallel, no contention. Enable in JPA via Hibernate hint: 
@QueryHints(@QueryHint(name = "jakarta.persistence.lock.timeout", value = "-2"))
Supported on Postgres, MySQL 8+, H2.

Sync send + tx rollback on failure

kafkaTemplate.send(record).get(10, TimeUnit.SECONDS);
e.setPublishedAt(Instant.now());

If the broker doesn't ack within 10s, the future throws → relay's @Transactional rolls back → none of the publishedAt updates from this batch commit → next poll retries the whole batch.

7.5 Delivery guarantee

At-least-once, NOT exactly-once.
  • The relay sends to Kafka, then marks published_at. If the JVM dies between broker ack and the DB commit, the row is re-sent next poll.
  • Consumers must be idempotent. Use the outbox-id header (or partition+offset) plus a processed_events table to skip duplicates.
True exactly-once across a DB and Kafka requires Kafka transactions + transaction-aware DB — much heavier and rarely needed.

7.6 Alternative: CDC (Debezium)

Instead of polling, stream the database WAL/binlog into Kafka with Debezium. Same effect — at-least-once events that are atomic with DB writes — without app code. Trade-off: a Kafka Connect cluster to operate.

7.7 Outbox vs direct publish — when to use each

This project demonstrates BOTH patterns side by side. EmployeeService uses outbox (OutboxPublisher + OutboxRelay). DepartmentService publishes directly via DepartmentEventPublisher. The right choice depends on what you'd lose if the event went missing.

Outbox (Employee)Direct publish (Department)
TriggerDB write (create / update / delete)DB read (findById) and DB write (create)
Atomic with DB tx?✅ Yes — outbox row commits iff business write commits❌ No — Kafka send is independent of any DB tx
Send modeSync .get(), retry on failure via tx rollbackAsync .whenComplete() with failure-logging callback
Latency added to API requestNone (relay runs on a separate thread)Tiny (just a non-blocking send() call)
If Kafka is down at the moment of writeEvent sits in outbox_events, retried until Kafka returns. Not lost.Event is logged and dropped. Lost.
Right for…Domain state changes — anything where consumers must see every event for correctness (read-model projections, cross-service workflows)Audit / observability / analytics — events that are useful but not load-bearing
Don't use direct publish for state changes. Department creation events here are direct-publish for simplicity, but in a real system that's wrong: a downstream consumer could miss "Department X was created" forever, and your read-model would be permanently stale. The outbox is the safe default; only deviate when the event is genuinely throwaway.

7.8 Multiple event types on one topic — dispatch pattern

Both employee.events and department.events carry multiple event types in a single topic (EmployeeCreated, EmployeeUpdated, EmployeeDeleted, EmployeePatched; DepartmentCreated, DepartmentLookedUp). The producer puts the type in a header; the consumer dispatches on it.

// Producer side (DepartmentEventPublisher.publish)
record.headers().add("event-type", eventType.getBytes(StandardCharsets.UTF_8));

// Consumer side (DepartmentEventListener)
String eventType = headerValue(record, "event-type");
switch (eventType) {
    case "DepartmentLookedUp" -> log.info("findByDeprtId was called — ...");
    case "DepartmentCreated"  -> log.info("Department created — ...");
    default                   -> log.warn("unknown event-type=...");
}

Why this beats one-topic-per-event-type: see § 3.1 (per-aggregate ordering, fewer topics, free new event types). The header-dispatch is a small extra branch in the consumer in exchange for those benefits.

8. Retention policy — where to set it

Kafka keeps messages for a configurable time/size. Once retention is exceeded, segments are deleted (or compacted).

8.1 The two flavors

PolicyBehaviorUse case
cleanup.policy=delete (default)Drop old segments based on time or sizeEvent streams (most cases)
cleanup.policy=compactKeep latest message per key, drop older"Current state" topics — change-data-capture, snapshots
cleanup.policy=compact,deleteBoth — compact AND age outChange-log with maximum retention

8.2 Where you can set it (most → least specific)

Per-topic at create time (most common)

kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic employee.events --partitions 3 --replication-factor 1 \
  --config retention.ms=604800000 \
  --config retention.bytes=10737418240 \
  --config cleanup.policy=delete

Per-topic on an existing topic

kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name employee.events \
  --alter --add-config retention.ms=259200000

From Spring Boot via KafkaAdmin

Declarative — Spring Kafka creates topics on startup if they don't exist:

            .partitions(3)

In server.properties or via env vars on the broker container. These are the fallback when a topic doesn't override.

Practical rule of thumb:
  • Time retention: 7 days for event topics, longer for audit/replay-on-bug-fix scenarios.
  • Size retention: cap to protect disk if events spike.
  • Both can coexist — Kafka deletes when EITHER limit is reached.

9.4 Failure scenarios — why this design holds

What failsWhat happensResult
Kafka down during Phase APhase A doesn't touch Kafka. DB tx commits. Outbox row sits with published_at=null.Phase B retries every 1s; publishes when Kafka returns. ✅

The whole point: every failure degrades to at-least-once — the strongest guarantee available without 2PC. No silent data loss, no ghost events. Duplicates are the consumer's problem, and they're a solvable problem (dedup table keyed by outbox-id).

10. Idempotency — making sure no duplicates leak through

At-least-once delivery means duplicates can happen. Three places, three answers — you need a strategy at each layer, but only the third actually requires you to write app code.

10.1 Layer 1 — producer-side duplicates

Cause: producer retries (transient broker errors) cause the same message to be appended twice on the broker.

Solution: enable.idempotence=true on the producer (already set in KafkaConfig). The broker assigns a producer id + sequence number to each message; duplicate sequence numbers from the same producer are rejected.

Status: ✅ already handled.

10.2 Layer 2 — outbox relay duplicates

Cause: relay sends to Kafka → broker acks → JVM dies before the DB UPDATE outbox_events SET published_at=now() commits. Next poll re-sends the same row.

Solution: cannot prevent this on the producer side — it's the inherent at-least-once trade-off. The fix lives in the consumer (Layer 3).

10.3 Layer 3 — consumer-side dedupe (the one you write)

Even with both layers above, a consumer can crash after processing a message but before its offset is committed. On restart it re-reads the message. So consumers that have side effects MUST dedupe.

The pattern

CREATE TABLE processed_events (
    event_id     VARCHAR(64) PRIMARY KEY,
    processed_at TIMESTAMP NOT NULL
);
@Transactional
public void onEvent(ConsumerRecord<String, String> record) {
    String eventId = headerValue(record, "outbox-id");
    if (processedEventRepo.existsById(eventId)) {
        log.info("Skipping duplicate eventId={}", eventId);
        return;
    }

    // ... do the real side-effect work ...

    processedEventRepo.save(new ProcessedEvent(eventId, Instant.now()));
}
Critical detail: the existsById check, the side-effect work, and the dedupe insert MUST run in the same DB transaction. Otherwise: process side effect → crash → dedupe row never written → next replay double-processes. Same dual-write problem flipped around.

Choosing the unique id

SourceWhen to use
outbox-id headerBest for outbox-driven flows (Employee). The relay sets this from the row's id; globally unique within the app.
Producer-generated UUID headerFor direct-publish flows (Department). Caveat: with async fire-and-forget there's no producer retry, so each call already produces one message — dedupe mostly insures against consumer-side replays.
Business keyBest when the message has natural identity (e.g. employee_id + version). Doubles as your idempotency token in business logic.
partition + offsetWorks for any topic, but breaks if you ever reset offsets and reprocess from the beginning.

Naturally idempotent operations — when you can skip the dedupe table

Side effectIdempotent?
UPDATE rows SET status = ? WHERE id = ?✅ Running twice is the same as once.
INSERT ... ON CONFLICT DO NOTHING (Postgres) / ON DUPLICATE KEY UPDATE (MySQL)✅ Built-in upsert idempotence.
SET balance = $X✅ Absolute set.
balance = balance + $X❌ Increment — must dedupe.
Send email / SMS / webhook❌ External side effect — must dedupe.
Append-only audit log❌ Two log lines for one event — must dedupe (or accept noise).
Design preference: make ops idempotent at the data layer when you can (upsert, absolute set). Reach for an explicit dedupe table only when the operation is inherently non-idempotent.

10.4 Operational hygiene

  • Bound the dedupe table size: it grows forever. Add a TTL/cleanup job (e.g. delete rows older than 7 days) once the broker can no longer replay older messages anyway.
  • Match the dedupe window to the retention window: if Kafka keeps messages 7 days and you dedupe with 7-day retention, you cover every possible replay.
  • Don't dedupe pure audit/log events: an extra log line is harmless. Save the dedupe-table cost for state-affecting handlers.

11.8 Composing common dev workflows

"Show me everything that's been published to the employee topic, key + headers"

docker exec -it spring-crud-kafka /opt/kafka/bin/kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 --topic employee.events \
  --from-beginning --max-messages 100 \
  --property print.key=true \
  --property print.headers=true \
  --property key.separator=" | "

"How far behind is my consumer?"

docker exec spring-crud-kafka /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 --describe --group spring-crud

Look at the LAG column — anything > 0 means messages are queued waiting to be consumed.

"Reset Spring's consumer to re-read every message from scratch"

# 1. Stop the Spring app first (consumer must be inactive):
lsof -ti:8080 | xargs kill -9 2>/dev/null

# 2. Reset all of spring-crud's offsets to earliest on both topics:
docker exec spring-crud-kafka /opt/kafka/bin/kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group spring-crud --reset-offsets \
  --topic employee.events --topic department.events \
  --to-earliest --execute

# 3. Restart the app:
./gradlew bootRun

"Wipe the broker and start fresh"

docker compose down -v
docker compose up -d

Removes all topics, all messages, all consumer-group offsets. The app will recreate topics on first publish.
